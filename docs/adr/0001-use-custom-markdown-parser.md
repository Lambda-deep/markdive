# ADR-0001: 軽量な自前Markdownパーサーの採用

**Status**: Accepted  
**Date**: 2026-03-28  
**Context**: Issue #3でコードブロック対応を検討する中で、Markdownパーサーライブラリ導入の是非を再評価

## Context（背景）

Issue #3「コードブロック内のMarkdown記法の誤検知を修正」への対応を検討する際、以下の2つのアプローチが考えられた：

1. **自前パーサーにコンテキスト追跡機能を追加** - コードブロックフラグを実装
2. **既存のMarkdownパーサーライブラリを導入** - `marked`, `remark`, `markdown-it` など

この決定は、md-dive の設計思想における優先順位を再確認するものである。

## Decision（決定内容）

md-dive は引き続き **自前のMarkdownパーサー**（正規表現ベース）を使用し、サードパーティのMarkdownパーサーライブラリは導入しない。

Issue #3のコードブロック対応は、パーサーにコンテキスト追跡ロジックを追加することで解決する。

## Rationale（理由）

### 要件定義書に明記された設計優先順位

[PR #1](https://github.com/Lambda-deep/md-dive/pull/1) の要件定義書（現在は `REQUIREMENTS.md`）には以下が明記されている：

> **依存性**: 最小限の依存関係（軽量なツールであること）。
> 重厚なMarkdownパーサー（Unified/Remark等）をラップするか、要件に特化した自前パーサーを採用する。

**軽量性**は md-dive のコア設計思想の1つであり、AIエージェントが高速にインストール・実行できることが求められる。

### 依存関係の比較

| 項目 | 自前パーサー | marked | remark | markdown-it |
|------|------------|--------|--------|-------------|
| 実行時依存 | 0 | 1 | 多数 | 1 |
| サイズ | ~5KB | ~40KB | ~100KB+ | ~100KB |
| npm install時間 | 最小 | 短い | 長い | 中程度 |
| 脆弱性リスク | 最小 | 低 | 中 | 低 |

現在の実行時依存は `commander` のみであり、この明快さを維持する価値がある。

### md-dive の特殊要件

md-dive には以下の特殊な要件があり、どのライブラリを使っても**カスタムロジックが必要**：

1. **階層IDの自動付与** - `1.2.3` 形式のID生成
2. **サマリーコメントの抽出** - `<!-- summary: ... -->` の認識

これらのロジックは自前パーサーでもライブラリでも同様に実装が必要である。

### CommonMark完全準拠の必要性

md-dive の対象は「開発者が書いた技術ドキュメント」であり：

- 極端なエッジケースへの対応は不要
- 主要な構文（見出し、コードブロック、引用、テーブル）の対応で十分
- 段階的に対応を拡充できる（Issue #3, #4）

## Consequences（結果）

### Good（良い影響）

- **軽量性の維持** - インストールサイズと速度が最小限
- **依存関係の単純さ** - `commander` のみという明快さ
- **脆弱性リスクの最小化** - 依存ライブラリ経由の脆弱性が入り込まない
- **要件に特化した実装** - 不要な機能を含まない

### Bad（悪い影響）

- **エッジケース対応は手動** - コードブロック、引用、テーブルなどの対応を個別実装
- **CommonMark完全準拠は困難** - すべての仕様への対応は現実的でない
- **メンテナンス負担** - Markdown仕様の変更への追従は手動

### Mitigation（緩和策）

- **テスト駆動開発** - Issue #3で提案されているように、フィクスチャとテストで対応状況を可視化
- **段階的拡充** - 必要な構文から優先的に対応
- **将来の見直し** - 保守コストが軽量性を上回った場合は再検討（新しいADRで判断）

## Alternatives Considered（検討した代替案）

### 案1: marked を採用

- **Pros**: 軽量（~40KB）、シンプルなAPI、高速
- **Cons**: 依存追加、カスタムロジックは結局必要
- **判断**: 軽量性の要件を満たすが、自前パーサーで対応可能

### 案2: remark（unified ecosystem）

- **Pros**: 拡張性が高い、プラグインエコシステム
- **Cons**: 依存が多数、サイズが大きい、学習コストが高い
- **判断**: md-dive の要件に対してオーバースペック

### 案3: markdown-it

- **Pros**: 高性能、プラグイン対応
- **Cons**: サイズが大きい（~100KB）、設定が複雑
- **判断**: 軽量性の要件に反する
