# ADR-0003: Agent Skill統合による構造的探索の標準化

**Status**: Accepted  
**Date**: 2026-03-29  
**Context**: 競合ツール（mdsel）との比較検討および、AIエージェントでの利用を前提とした標準導線の再設計

## Context（背景）

大規模MarkdownをAIエージェントが扱う際、全文読みによるトークン消費増大と、文書中盤の情報埋没（Lost in the Middle）が課題となる。

競合である `mdsel` は「必要な見出しを既に知っている」前提での抽出に強みがある一方、`markdive` は「どこに何があるか分からない状態」から構造を辿って探索するユースケースに強みを持つ。

本プロジェクトでは、以下の設計思想を維持する必要がある。

- 軽量依存・高速動作（実行環境を選ばない）
- 人間と同様の論理構造に基づく段階的理解
- セクション単位の文脈維持（見出し階層とパンくず）

この前提のもと、AIエージェントに `markdive` を標準ツールとして使わせるための統合方針を定義する。

## Decision（決定内容）

以下を採用する。

1. **統合方式は「Agent Skill + CLI」を採用し、MCPサーバーは採用しない。**
2. **AI利用時の基本導線を `dive -> read` に固定する。**
3. **構造を壊す機能は非採用とする。**
4. **Agent Skill向けの運用ガイダンスを整備し、行動変容を促す。**

具体的には次の通り。

### 採用するもの

- `SKILL.md` による段階的ロード前提のガイド（構造把握を先行させる手順）
- `dive` での段階的開示、`read` での局所精読という責務分離
- `read` 出力の現在地情報（source/path/context）を維持
- エージェント設定側で利用できるフックテンプレート（例: 大きなMarkdownでの `Read` 後に `markdive` 利用を促すメッセージ）をドキュメントで提供

### 現時点で非採用にするもの

- MCPサーバー実装（常時ツール定義注入によるオーバーヘッドを避けるため）
- キーワード直ジャンプ中心の探索モデル
- 行数ベースの物理切り出し（head/tail相当）
- 埋め込みモデル前提のセマンティック検索（RAG）
- 編集機能の取り込み（本ツールは読み取り・探索に特化）

### 今後の拡張候補（軽量性を損なわない範囲）

- `read` 後に次に読む候補ID（親・兄弟・子）を提示する補助メタデータ
- サマリー抽出ヒューリスティクスの改善（箇条書き優先など）
- `dive` 時のセクションサイズ情報（文字数や推定トークン）

## Rationale（理由）

- `markdive` の価値は「正解を知っている前提の抽出」ではなく、「未知文書の構造的ナビゲーション」にある。
- `dive -> read` は、人間の読解手順（目次把握してから本文精読）と一致し、エージェントの推論を安定化しやすい。
- 軽量依存を維持することで、導入・実行コストを最小化し、あらゆる環境での再現性を確保できる。
- 検索・行数切り出しを主軸にすると、周辺文脈の欠落により誤読リスクが上がる。
- MCPは有力な統合手段だが、本プロジェクトの優先順位（薄い依存・軽量動作）とは現時点で整合しない。

## Consequences（結果）

この決定によって、AI向けの利用導線を明確化しつつ、既存設計思想を維持できる。

**Good（良い影響）**:

- AIエージェントでの利用手順が標準化される（`dive -> read`）
- 非アクティブ時の余計なオーバーヘッドを抑えられる
- 文書の現在地を見失いにくく、推論時の接地性が高まる
- 依存関係を増やさず、ツールの軽量性を維持できる

**Bad（悪い影響）**:

- ピンポイント抽出型ツールより到達ターン数が増えることがある
- Agent Skillやフックは呼び出し側環境への設定依存がある
- 高度検索（意味検索など）を期待する用途には適合しにくい

**Mitigation（緩和策）**:

- `SKILL.md` とREADMEで最短操作例を明示し、探索ターンを削減する
- フック設定のテンプレートを提供し、導入障壁を下げる
- 軽量な範囲の補助機能（候補ID提示、統計表示）を段階的に検証する

## Alternatives Considered（検討した代替案）

### 案1: MCPサーバーを公式統合として採用

- **Pros**: ツール呼び出しの統一性が高い
- **Cons**: 常時スキーマ注入によるコスト、実装・運用負荷
- **判断**: 現時点では軽量性の優先に反するため非採用

### 案2: mdsel型の抽出中心モデルに寄せる

- **Pros**: 既知パスへの到達が速い
- **Cons**: 未知文書探索や周辺文脈の保持に不利
- **判断**: `markdive` の主目的と異なるため非採用

### 案3: セマンティック検索（RAG）を導入

- **Pros**: 意味ベース検索が可能
- **Cons**: 重い依存と運用コスト、再現性低下
- **判断**: 軽量性と可搬性を損なうため非採用
